<precision_model>
This is the load-bearing reasoning the implementation MUST encode. Read before writing any code.

1. UniVec3 stores position as `Units(Long3 integer) * Scale + Offset(Double3)`, expressed in the object's PARENT frame (`UniObject.LocalPos` is relative to its parent). Integer Units give exact large-scale precision; Offset gives sub-cell double precision.

2. Subtracting two UniVec3 AT THE SAME Scale is exact: `operator-` does `a.Units - b.Units` (exact 64-bit integer subtraction, regardless of magnitude) plus `a.Offset - b.Offset` (double). This is the precision superpower: exact relative vectors even at intergalactic separation.

3. THE TRAP: `UniVec3.Convert(newScale)` and cross-scale `operator-/operator+` (via `EnsureSameScale → Convert`) route through `ToDouble3()` (`Units*Scale + Offset`), which forms an ABSOLUTE-metres Double3 and zeroes Units. For SOI-BOUNDED single values (one LocalPos, ~1e9–1e15 m) this is harmless (fits in double). For ABSOLUTE-FROM-ROOT accumulated values (up to ~1e30 m at Universe scale 1e16 m/unit) it loses low bits and, when differenced, causes catastrophic cancellation. So: converting ONE LocalPos up ONE level is fine; forming an absolute-from-root coordinate then subtracting is NOT.

4. CORRECT strategy (proven last task in METRES; this task redoes it in UNIVEC3):
   - Find the lowest common ancestor (LCA) of the two objects.
   - Bring BOTH objects' positions into the LCA's child-frame as UniVec3 by walking child→parent, applying the per-level pattern `pos = pos.Convert(parentScale) + parent.LocalPos` (the SAME pattern as `GameWorld.ChildPosToParentSpace`). Each step is SOI-bounded → precise. Stop AT the LCA — never walk to Root.
   - Why Units survive the walk: each `Convert(parentScale)` zeroes Units into Offset, but the subsequent `+ parent.LocalPos` constructor re-runs `Normalize()`, which splits the offset back into integer Units. So after every step the operand again holds proper integer Units. At the LCA frame both operands share the LCA child-scale.
   - Subtract the two SAME-SCALE UniVec3 in the LCA frame → exact integer Units cancellation. The huge common-ancestor offset is never formed.
   - ONLY THEN call `ToDouble3()` on the small resulting delta to get metres for rendering/normalization. The metres conversion happens EXACTLY ONCE, on a small differenced vector.

5. DO NOT change the semantics of `UniVec3.Convert` / `operator-` / `GameWorld.ChildPosToParentSpace` — they are correct for their SOI-bounded callers (SOI transitions). Build UniMath ON TOP of them. Prefer putting orchestration in UniMath; add a tiny pure helper to UniVec3 only if clearly warranted.

EQUIVALENCE AT CURRENT MVP SCALE (why behaviour is unchanged):
The MVP scene (TestSetup) hierarchy: Root→Galaxy(Universe space)→Star(Galaxy space)→PlanetA/PlanetB(Star space)→Ship(Planet space); siblings ALPHA CEN/BARNARD/SIRIUS live directly under Galaxy in Galaxy space. The ship's NextTierSkybox bodies are the Galaxy-space siblings. LCA(ship, sibling) = Galaxy (index 1). The Galaxy sits at the origin of Root, so the LCA-frame offsets are all SOI-bounded (≤ ~1e17 m, well within double) — exactly the regime the prior metres walk already handled correctly. Therefore UniMath produces the same metres delta (within float tolerance). The WIN is precision HEADROOM at true Universe scale, not a behaviour change at MVP scale.
</precision_model>

Implement exactly these members:

1. `public static int FindLca(UniObject a, UniObject b, List<UniObject> objs)` — port the proven algorithm from SkyboxRenderer.FindLca verbatim in behaviour: build a HashSet of a's ancestor indices (including a.Index) by walking ParentIndex, then walk b's chain upward returning the first index in that set; -1 if none. Keep the `(uint)idx < (uint)objs.Count && objs[idx] != null` guard and the `parentIdx == idx` cycle guard on BOTH walks. Hierarchy depth ≤ 5.

2. `public static UniVec3 ToAncestorFrame(UniObject node, int ancestorIdx, List<UniObject> objs)` — return node.LocalPos expressed in the ancestor's child-frame as a UniVec3, accumulated IN UNIVEC3 (this is the key difference from the old metres-based PositionRelativeToAncestor). Algorithm: if `node.Index == ancestorIdx`, return a zero vector at the ancestor's child-scale (i.e. `new UniVec3(Double3.Zero, node.LocalPos.Scale)` is wrong here — the node IS the ancestor so its child-frame is one level DOWN; return `new UniVec3(Double3.Zero, UniObject.Scale(node.CurrentSpace))` representing "at the ancestor origin"). Otherwise start with `UniVec3 pos = node.LocalPos;` then walk parents: `int pIdx = node.ParentIndex; while ((uint)pIdx < (uint)objs.Count && objs[pIdx] != null) { if (pIdx == ancestorIdx) break; var parent = objs[pIdx]; double parentScale = UniObject.Scale(parent.CurrentSpace); if (parentScale <= 0) break;  // Root has Scale -1; never multiply Root in  pos = pos.Convert(parentScale) + parent.LocalPos; pIdx = parent.ParentIndex; }` then `return pos;`. This reuses the exact `GameWorld.ChildPosToParentSpace` per-level pattern (`childPos.Convert(parentScale) + parent.LocalPos`). The result is in the ancestor's child-frame scale (the scale of the LCA's children), with integer Units intact after each Normalize. Each step is SOI-bounded → precise (precision_model point 4). Keep the `(uint)`+null loop guard and the `parentScale <= 0` Root guard.

3. `public static bool RelativePosition(UniObject from, UniObject to, List<UniObject> objs, out UniVec3 result)` — the PRIMARY precision primitive. `int lca = FindLca(from, to, objs); if (lca < 0) { result = new UniVec3(Double3.Zero, ...); return false; }` then `UniVec3 toFrame = ToAncestorFrame(to, lca, objs); UniVec3 fromFrame = ToAncestorFrame(from, lca, objs); result = toFrame - fromFrame; return true;`. Both operands are at the SAME LCA child-scale (both produced by ToAncestorFrame to the same ancestor), so `operator-` performs EXACT integer Units cancellation — the large common-ancestor offset is never formed (precision_model points 2 + 4). Do NOT call ToDouble3 here. For the false branch pick a documented zero sentinel scale (e.g. `UniObject.Scale(UniObject.ChildSpace(UniObject.Space.Root))` or simply `1.0`); document that callers treat false as the coincident/no-common-ancestor case.

4. `public static Double3 RelativeMetres(UniObject from, UniObject to, List<UniObject> objs)` — the ONE sanctioned metres conversion: `return RelativePosition(from, to, objs, out var rel) ? rel.ToDouble3() : Double3.Zero;`. ToDouble3 is applied only to the small differenced vector.

5. `public static double Distance(UniObject a, UniObject b, List<UniObject> objs)` — `=> RelativeMetres(a, b, objs).Magnitude();`.

Add a class-level XML-doc summary stating: read-only consumer contract; the LCA-relative UniVec3 strategy; and a WARNING block mirroring SkyboxRenderer's existing one — "do NOT replace ToAncestorFrame's per-level Convert+add with a naive cross-scale `to - from`: cross-scale operator- routes through Convert→ToDouble3 which zeroes Units and collapses absolute-from-root values to a single double, reintroducing catastrophic cancellation." Reference the precision_model reasoning inline so it survives in-file.

Confirm callers' frame semantics match: `RelativePosition(from, to)` returns `to − from` in the LCA frame (i.e. the vector FROM `from` TO `to`). SkyboxRenderer needs body−ship (vector from ship to body) → call `RelativePosition(ship, body, ...)`. Document the direction convention in the XML-doc precisely so Task 2 wires it correctly.
