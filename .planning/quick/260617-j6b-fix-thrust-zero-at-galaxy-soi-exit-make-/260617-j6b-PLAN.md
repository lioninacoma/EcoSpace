<objective>
Fix the `thrust-zero-at-galaxy-soi-exit` tech debt: `UpdateSpeedEnvelope` in
`Scripts/Flight/FlightController.cs` clamps `targetMax` by proximity to the nearest
body's surface with no regard for direction of travel. At a galaxy SOI edge the ship
is "close" to the galaxy, so the envelope collapses `targetMax` toward `MinSpeed`.
That ease-in is correct on APPROACH but wrongly fires on EXIT, stranding the player
at the boundary with near-zero thrust.

Make the proximity clamp DIRECTION-AWARE: track the single nearest body during the
scan, compute the radial closing velocity (dot of the ship's motion direction with the
unit direction toward that body). When RECEDING (closing <= 0), exempt the proximity
clamp so `targetMax` ramps back to `MaxSpeed` via the existing lerp. When APPROACHING
(closing > 0), keep the current ease-in exactly.

Purpose: restore symmetric, smooth flight at galaxy SOI boundaries without adding an
FTL mode (D-35/D-36 constraint: this stays a tuning/envelope change).
Output: revised `UpdateSpeedEnvelope`; tech-debt todo closed.
</objective>

  <name>Task 1: Make the proximity clamp direction-aware in UpdateSpeedEnvelope</name>
  <files>Scripts/Flight/FlightController.cs</files>
  <action>
Edit `UpdateSpeedEnvelope`. During the nearest-body scan (the parent-body branch AND the
sibling loop), track the SINGLE body that currently holds the minimum surface distance:
when a candidate becomes the new `nearest`, also record the unit direction from the ship
TOWARD that body's centre, in the parent meters frame as a Double3.
  - For the PARENT body: toward-direction = -normalize(ship.LocalPos in meters). Compute the
    meters vector via ship.LocalPos.ToDouble3(); negate and normalize it (divide by its
    Magnitude()). Guard against a zero magnitude (ship exactly at centre) — skip recording a
    direction in that degenerate case.
  - For a SIBLING body: toward-direction = normalize((body.LocalPos - ship.LocalPos) in meters).
    Form the delta UniVec3 (body.LocalPos - ship.LocalPos), call ToDouble3() once on it, then
    normalize (guard zero magnitude). Both share the parent frame so this is exact.

After the scan, before computing `targetMax`, derive the ship's motion direction in the
parent meters frame: motionDir = (-_shipBasis.Z) * sign(_throttle01), built as a Double3 from
the Vector3 components. Then closing = Double3.Dot(motionDir, towardNearestUnit).
  - If `_throttle01` is ~0 (abs < small epsilon, e.g. 1e-6) OR no toward-direction was recorded
    (degenerate/open-space case where `nearest` came from the open-space fallback): treat as
    APPROACHING (keep the proximity clamp). This is safe because actualSpeed = throttle * contextMax
    is ~0 anyway, and it avoids normalizing a zero/undefined vector.
  - If closing > 0 → APPROACHING: keep the existing clamp exactly:
    targetMax = Mathf.Clamp(nearest * _speedPerMeter, _minSpeed, _maxSpeed).
  - If closing <= 0 → RECEDING: EXEMPT the proximity clamp — set targetMax = _maxSpeed
    (treat as open space) so the existing _contextMax lerp ramps it back up smoothly.

Do NOT change the open-space fallback (`nearest == double.MaxValue` branch), the
`_contextMax` lerp, the `targetSpeed = _throttle01 * _contextMax` line, or the `_easedSpeed`
lerp — both lerps must still run on every path so there is no speed snap on approach OR exit
(D-07). Do NOT add a new flight mode, new [Export] knobs, or touch any other method. Update
the `UpdateSpeedEnvelope` XML doc-comment to note the direction-aware (receding-exempt)
clamp and reference the tech-debt fix.
