## Critical Anti-Patterns

| Pattern | Description | Severity | Prevention Mechanism |
|---------|-------------|----------|---------------------|
| Stale scale-dependent cache across SOI | `WorldRenderer` (then `RenderBridge`) cached each body's mesh radius at creation time; it never updated when `ship.LocalPos.Scale`/render factor changed on an SOI transition, so a planet sized for Planet space rendered ~10,000× too large in Star space (the "giant blue ball"; camera ended up *inside* the sphere → planet looked "gone"). | advisory | Any value derived from the current-frame scale/factor (radius, etc.) MUST be recomputed every frame in `RenderBodyAt`, exactly like positions — never baked once in `GetOrCreateMesh`. |
| Godot Control mouse_filter swallows motion | `Hud`/`Crosshair`/`SteeringReticle` Controls used the default `mouse_filter = Stop`, consuming `InputEventMouseMotion` before `_UnhandledInput` ran, so mouse steering silently did nothing. | advisory | Set `mouse_filter = Ignore` on HUD Controls AND read mouse accumulation in `_Input` (runs before GUI dispatch, cannot be blocked). |
| Per-space light-type inconsistency | Lighting bodies with an OmniLight in Star space but a DirectionalLight in Planet space made the day/night shading *character* change across the SOI boundary. | advisory | Body lighting is now done in a single `unshaded` shader (`body_lit.gdshader`) computing Lambert from the nearest-star direction — identical in every space. Do not reintroduce Godot light nodes for bodies. |
