## Phase Goal

**As a** pilot, **I want to** fly my ship through a star system and approach dithered planets and a star while reading my speed and surroundings on a retro HUD, **so that** I can navigate a true 1:1 cosmos that stays correct and feels good at every scale.

> Note: derived from the ROADMAP prose goal ("Player can fly around a single star system, approach dithered planets and stars, and read their speed and context on a minimal HUD") — the goal is unambiguously player-framed, so it is expressed here as the canonical user story for MVP planning.

## Walking Skeleton Slice

This is the thinnest end-to-end working slice that proves the full stack against the existing precision engine: **input → simulation (UniVec3/SOI, hardened) → floating-origin render → HUD**. It deliberately uses a single placeholder thrust key (not the full flight model) and a single un-colored body — flight feel, body look, and full HUD are later slices. After this plan, a real user can launch the game, thrust, see one body grow, and watch a live speed number — none of which is possible today (the scene currently runs a fixed autopilot to Planet B and renders a static sphere).

<objective>
Establish the Walking Skeleton: harden `GameWorld.TrySpaceTransition` into an iterative, null-safe loop (STAB-01); add a `RenderBridge` node that holds the ship at the floating origin and renders only current-parent-space bodies ship-relative (RND-01/RND-02); replace the `TestSetup` autopilot with a minimal keyboard thrust that drives `TranslatePos` (WORLD-01 base, TRV-01 partial); and add a phosphor-green HUD label showing a live raw speed value (HUD-01 partial).

Purpose: Prove the input→sim→render→HUD loop works crash-free on the locked `UniVec3`/SOI engine before layering flight feel, body look, and the full HUD.
Output: Hardened `GameWorld`, new `RenderBridge` + `Hud` nodes wired into `Main.tscn`, `InputMap` thrust actions, and a runnable star-system flight skeleton.
</objective>

<threat_model>
## Trust Boundaries

EcoSpace is a single-player, fully offline desktop game: no network, no auth, no persistence, no untrusted external input. The only material robustness ("security") surface is internal state corruption / crash safety in the SOI transition machinery — exactly what STAB-01 mandates (RESEARCH Security Domain).

| Boundary | Description |
|----------|-------------|
| flight input → `GameWorld.TranslatePos` | Per-frame `Double3` delta crosses into the simulation; a large/NaN delta can drive multi-SOI transitions in one frame |
| `GameObjects` index references → object dereference | `ParentIndex`/`ChildIndices`/`siblingIndex` are integer indices that may point at null slots (after a future RemoveGameObject) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Denial of Service (local) | `TrySpaceTransition` recursion | mitigate | Replace recursion with a bounded `while` loop (`MaxIterations=32`); a math oscillation logs an anomaly and breaks instead of stack-overflowing/hanging (Task 1) |
| T-01-02 | Tampering / crash | `GameObjects[i]` dereference in transition family | mitigate | Route every lookup through null-safe `Get(i)`; bail to `false` on null (Task 1) — prevents NullReferenceException cascades (CONCERNS.md) |
| T-01-03 | Tampering | `ChildIndices` mutated mid-`foreach` | mitigate | Keep "find first → mutate → return" shape (exits enumerator) or snapshot before iterating (Task 1) — prevents `InvalidOperationException: Collection was modified` |
| T-01-04 | Denial of Service (visual) | far-plane / single-precision overflow at large coords | mitigate | Floating origin keeps render coords ship-relative and small (RND-01, Task 2); render only in-current-space bodies (RND-02, Task 2) |
| T-01-05 | Denial of Service (local) | oscillating exit/enter at SOI boundary | mitigate | Carry `lastExitedIndex` as `excludeIndex` so the ship does not re-enter the just-exited SOI in the same frame (Task 1, Pitfall 1) |

No network, auth, session, secrets, or cryptography surface exists in this phase (RESEARCH Security Domain: ASVS V2/V3/V4/V6 = N/A; V5 input validation applies only as crash-safety on transition indices, covered by T-01-02).
</threat_model>
