## Phase Goal

**As a** pilot, **I want to** fly my ship through a star system and approach dithered planets and a star while reading my speed and surroundings on a retro HUD, **so that** I can navigate a true 1:1 cosmos that stays correct and feels good at every scale.

## MVP Slice — Body Rendering

This slice makes the world *look* like the target: each body gets an authored hue and its true 1:1 radius, the star becomes a blooming emissive light source that illuminates planets with a real terminator, and the dither post-process is extended so distinct hues survive the 8-bit quantize. After this plan, a real user can visually distinguish Earth-blue Planet A, Mars-rust Planet B, and the yellow Star — and watch them grow from specks on approach — which the skeleton (single un-colored body) could not show.

<objective>
Author per-body presentation data (`Name`, `BaseColor`, `RadiusMeters`) on `UniObject` and set it for each body in `SetupScene` (WORLD-01 extension); make `RenderBridge` apply per-body albedo color and true 1:1 radius scaling, render the star as an unshaded emissive sphere with a glow `Environment` and an `OmniLight3D` (no shadows) that lights planets (RND-03/RND-04/D-13/D-14/D-16); and extend `dithering.gdshader` from its current 1-bit black/white output to a hue-preserving palette/per-channel ordered dither so bodies stay distinguishable by color (D-13, resolved by user).

Purpose: Deliver the retro dithered 1:1 body look that makes the star system legible and atmospheric, and provide the `RadiusMeters` data that Plan 03's distance-scaled speed depends on.
Output: Extended `UniObject` + authored scene data, color/emissive/lighting in `RenderBridge`, and a color-capable dither shader.
</objective>

## Trust Boundaries

Same single-player offline posture as Plan 01 — no network/auth/persistence/untrusted input. This plan's surface is purely rendering/asset robustness; the only failure modes are visual corruption or a missing-asset crash, not a security compromise.

| Boundary | Description |
|----------|-------------|
| shader asset path → `UniRenderer` load | `dithering.gdshader` is loaded by hardcoded path with no fallback (CONCERNS.md) |
| 1:1 body coordinates → single-precision render transform | Bodies at honest 1:1 radii/distances can push render math toward precision/far-plane limits if rendered out-of-space |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Denial of Service (visual) | OmniLight range/attenuation at 1:1 distances | mitigate | Light authored in the ship-relative render frame (small meters), `[Export]` range/energy tuned to cover the rendered scene (Task 2, Pitfall 7) — wrong values degrade visuals only, never crash |
| T-02-02 | Denial of Service (visual) | far-plane / precision overflow from rendering out-of-space bodies | mitigate | RenderBridge renders only `parent.ChildIndices` (RND-02, inherited from Plan 01); 1:1 specks stay within the floating-origin render range |
| T-02-03 | Denial of Service (local) | hardcoded shader path with no fallback (CONCERNS.md) | accept | Pre-existing engine behavior; this plan extends the shader in place and does not move it, so the existing load path is unchanged. Out-of-scope to add a fallback loader (deferred tech debt). |
| T-02-04 | Tampering | `quantize_levels` uniform reaching 0 → divide-by-zero / NaN in shader | mitigate | `UniRenderer` guards the export with `Mathf.Max(1, value)` (Task 3) |

No network, auth, session, secrets, or cryptography surface in this phase.
