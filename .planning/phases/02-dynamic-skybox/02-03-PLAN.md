<task type="checkpoint:decision" gate="blocking">
  <decision>Test framework + compilation strategy for the standalone test project</decision>
  <context>
    `TierClassifier.Classify(UniObject body, UniObject ship)` references `UniObject`, and `UniObject` imports `using Godot;` and uses `Godot.Color`. A truly Godot-free test project therefore cannot compile `UniObject` without a Godot type reference. Two decisions are coupled: (a) which test framework, and (b) how to give the test project access to the types under test. The RESEARCH.md test-framework versions (gdUnit4.api 5.0.0 / gdUnit4.test.adapter 3.0.0) are tagged [ASSUMED] and not verified on NuGet.
  </context>
  <options>
    <option id="xunit-godotsharp-linked">
      <name>xUnit + GodotSharp PackageReference, link source files (RECOMMENDED)</name>
      <pros>xUnit versions are well-known and stable (no [ASSUMED] package, no legitimacy gate). `GodotSharp` is a published NuGet package that supplies `Godot.Color` so `UniObject` compiles. Linking `TierClassifier.cs` + `UniObject.cs` via `<Compile Include="..\Scripts\...">` avoids a project reference to the Godot SDK project. `dotnet test` works from CLI.</pros>
      <cons>Pulls GodotSharp into the test project (heavier than a pure classlib); must keep the linked file list correct if UniObject gains new Godot dependencies.</cons>
    </option>
    <option id="gdunit4">
      <name>gdUnit4.api + gdUnit4.test.adapter</name>
      <pros>Godot-native test framework; integrates with the Godot editor test runner.</pros>
      <cons>Package versions are [ASSUMED] (RESEARCH.md A1/A2) and MUST be verified on nuget.org before install — requires a blocking legitimacy checkpoint. Heavier setup than needed for pure-logic tests.</cons>
    </option>
    <option id="extract-godot-free-core">
      <name>Refactor a Godot-free classification core, pure classlib + xUnit</name>
      <pros>Test project needs no GodotSharp at all (cleanest dependency graph).</pros>
      <cons>Requires changing the Plan 01 classifier signature to take primitives (e.g. `Space` + bool) instead of `UniObject`; `Space` is nested in the Godot-importing `UniObject` so even the enum drags the type in unless also extracted — larger refactor for marginal benefit.</cons>
    </option>
  </options>
  <resume-signal>Select: xunit-godotsharp-linked (recommended), gdunit4, or extract-godot-free-core. If gdunit4, a package-legitimacy verification checkpoint will be inserted before install.</resume-signal>
</task>
